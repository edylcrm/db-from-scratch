В предыдущих главах мы построили append-only log с hash index. Это работает, но у подхода два
фундаментальных ограничения: все ключи должны поместиться в RAM и нет поддержки range scan.

LSM-tree (Log-Structured Merge Tree) решает обе эти проблемы. Это архитектура storage engine,
которая стала стандартом де-факто для write-intensive баз данных и лежит в основе LevelDB,
RocksDB, Pebble, Cassandra, ScyllaDB, HBase и десятков других систем.

В этой главе мы разберём, как LSM-tree работает на концептуальном уровне — прежде чем в следующих
главах реализовывать каждый компонент по отдельности.

## Три кирпичика

LSM-tree состоит из трёх основных компонентов, которые работают в связке:

```
                          ┌─────────────────────────┐
   Put(key, value) ──────►│     WAL (диск)          │
                          │  Append-only log        │
                          │  Durability гарантия    │
                          └────────────┬────────────┘
                                       │
                                       ▼
                          ┌─────────────────────────┐
                          │    MemTable (память)    │
                          │    Skip List            │
                          │    Отсортированные KV   │
                          └────────────┬────────────┘
                                       │ flush (когда заполнится)
                                       ▼
                   ┌──────────────────────────────────────┐
                   │          SSTables (диск)             │
                   │                                      │
                   │  ┌─────────┐ ┌─────────┐ ┌────────┐  │
                   │  │ SST #1  │ │ SST #2  │ │ SST #3 │  │
                   │  │ (новый) │ │         │ │(старый)│  │
                   │  └─────────┘ └─────────┘ └────────┘  │
                   │                                      │
                   │ Иммутабельные, отсортированные файлы │
                   └──────────────────────────────────────┘
```

### WAL (Write-Ahead Log)

WAL — это тот самый append-only log, который мы уже написали. Но в LSM-tree у него другая роль:
WAL не является основным хранилищем, а служит страховкой для memtable.

Memtable живёт в памяти. Если процесс упал — данные в memtable теряются. WAL гарантирует, что этого
не произойдёт: каждая запись сначала попадает в WAL на диске, и только потом — в memtable.
При восстановлении после краша мы проигрываем WAL и восстанавливаем memtable.

Запись в WAL — это один sequential append, самая быстрая операция, которую можно сделать с диском.
Как только memtable сбрасывается на диск в виде SSTable, соответствующий WAL можно удалить.

### MemTable (Skip List)

Memtable — это отсортированная структура данных в памяти, куда попадают все записи. В отличие от hash table,
memtable поддерживает упорядоченный обход — это даёт нам range scan.

В качестве memtable обычно используется **skip list** — вероятностная структура данных,
которая обеспечивает O(log n) на вставку, поиск и удаление, а также эффективный последовательный обход.
Skip list проще в реализации, чем сбалансированные деревья (красно-чёрные, AVL),
и хорошо работает в concurrent-сценариях.

Когда memtable достигает заданного размера (обычно несколько мегабайт), она «замораживается» — становится
immutable — и сбрасывается на диск как SSTable. Одновременно создаётся новая пустая memtable для приёма записей.
Замороженная memtable остаётся доступной для чтения, пока flush не завершится.

### SSTable (Sorted String Table)

SSTable — это отсортированный иммутабельный файл на диске, в который сериализуется содержимое memtable.
«Иммутабельный» — ключевое слово: после создания SSTable никогда не модифицируется.

SSTable — это не просто плоский список пар ключ-значение. Внутри он имеет структуру:

```
  ┌──────────────────────────────────────────────┐
  │              SSTable файл                    │
  │                                              │
  │  ┌────────────────────────────────────────┐  │
  │  │         Data Blocks                    │  │
  │  │  Отсортированные KV-пары,              │  │
  │  │  разбитые на блоки                     │  │
  │  └────────────────────────────────────────┘  │
  │  ┌────────────────────────────────────────┐  │
  │  │         Index Block                    │  │
  │  │  Ключ → offset блока данных            │  │
  │  │  (позволяет найти нужный блок          │  │
  │  │   без сканирования всего файла)        │  │
  │  └────────────────────────────────────────┘  │
  │  ┌────────────────────────────────────────┐  │
  │  │         Bloom Filter                   │  │
  │  │  «Есть ли ключ в этом SSTable?»        │  │
  │  │  Быстрый ответ без чтения данных       │  │
  │  └────────────────────────────────────────┘  │
  │  ┌────────────────────────────────────────┐  │
  │  │         Footer                         │  │
  │  │  Offsets index block и bloom filter    │  │
  │  └────────────────────────────────────────┘  │
  └──────────────────────────────────────────────┘
```

Благодаря index block чтение из SSTable — это бинарный поиск по индексу + чтение одного блока данных,
а не сканирование всего файла. Bloom filter позволяет быстро определить, что ключа точно нет в этом SSTable,
и пропустить его вообще.

## Путь записи

Каждая операция записи проходит два шага:

```
  Put("user:42", "{name: alice}")

  Шаг 1: WAL append (диск)
  ┌──────────────────────────────────────────┐
  │ WAL: [...] [user:42 → {name: alice}]     │
  │                                     ▲    │
  │                         sequential write │
  └──────────────────────────────────────────┘

  Шаг 2: MemTable insert (память)
  ┌──────────────────────────────────────────┐
  │ MemTable (skip list):                    │
  │   config   → ...                         │
  │   user:1   → ...                         │
  │   user:42  → {name: alice}  ← вставили   │
  │   user:99  → ...                         │
  └──────────────────────────────────────────┘
```

Оба шага — быстрые. WAL append — sequential write на диске. Вставка в skip list — O(log n) в памяти.
Никаких random I/O.

Когда memtable заполняется:

```
  MemTable заполнилась (размер > порог)
  │
  ├─ 1. Замораживаем текущую memtable (immutable)
  │
  ├─ 2. Создаём новую пустую memtable + новый WAL
  │
  ├─ 3. В фоне: сбрасываем immutable memtable → SSTable
  │     (записи уже отсортированы — один sequential write)
  │
  └─ 4. Удаляем старый WAL (данные теперь в SSTable)
```

## Путь чтения

Чтение устроено сложнее. Нужно проверить несколько мест в определённом порядке:

```
  Get("user:42")

  1. MemTable (активная)      ← самые свежие данные
     │ Нашли? → возвращаем
     │ Не нашли ↓
     │
  2. Immutable MemTable        ← данные, которые сейчас сбрасываются
     │ Нашли? → возвращаем
     │ Не нашли ↓
     │
  3. SSTable #1 (самый новый)  ← проверяем bloom filter, потом index
     │ Нашли? → возвращаем
     │ Не нашли ↓
     │
  4. SSTable #2
     │ Нашли? → возвращаем
     │ Не нашли ↓
     │
  5. SSTable #3 (самый старый)
     │ Нашли? → возвращаем
     │ Не нашли ↓
     │
  6. ErrNotFound
```

Порядок важен: мы идём от новых данных к старым. Если ключ был обновлён — первая найденная версия будет актуальной.

Bloom filter на каждом SSTable критически важен: он позволяет пропустить файлы, в которых ключа заведомо нет,
не читая с диска. Без bloom filter чтение несуществующего ключа требовало бы проверки каждого SSTable.

## Compaction

Со временем SSTable'ов становится много. Чем больше файлов — тем медленнее чтение (больше мест для проверки).
Compaction решает эту проблему: фоновый процесс сливает несколько SSTable'ов в один,
удаляя устаревшие версии ключей и tombstone.

```
  До compaction:
  ┌─────────┐ ┌─────────┐ ┌─────────┐
  │ SST #1  │ │ SST #2  │ │ SST #3  │
  │ a→1     │ │ a→2     │ │ b→1     │
  │ c→1     │ │ b→DEL   │ │ c→1     │
  └─────────┘ └─────────┘ └─────────┘

  Compaction (merge sort по ключам, берём последнюю версию):
  a: SST#1 имеет a→1, SST#2 имеет a→2 → берём a→2 (новее)
  b: SST#2 имеет b→DEL (tombstone) → удаляем
  c: SST#1 имеет c→1 → берём c→1

  После compaction:
  ┌─────────┐
  │ SST new │
  │ a→2     │
  │ c→1     │
  └─────────┘
```

Три файла превратились в один. Мёртвые данные удалены. Чтение стало быстрее.

Существуют разные стратегии compaction (size-tiered, leveled), но их мы разберём позже.
Главное сейчас — понять, зачем compaction нужен.

## LSM-tree vs B-tree

B-tree — главная альтернатива LSM. Это проверенная структура, которая лежит в основе
PostgreSQL, MySQL (InnoDB), Oracle, SQL Server. B-tree существует с 1971 года и до сих
пор остаётся стандартом для реляционных баз.

Ключевое различие: B-tree обновляет данные по месту (перезаписывает страницу на диске), а LSM-tree
всегда дописывает новые данные и периодически сливает файлы.

```
  B-tree: обновление ключа = random write в нужную страницу
  ┌─────┐
  │page │ ← перезаписали значение на месте
  └─────┘

  LSM-tree: обновление ключа = sequential append
  [...старые данные...][новое значение] ←
```

Это приводит к разным компромиссам:

| | LSM-tree | B-tree |
|---|---|---|
| **Запись** | Быстрая: sequential append | Медленнее: random I/O (поиск + перезапись страницы) |
| **Чтение (point lookup)** | Медленнее: возможно несколько SSTable | Быстрая: один путь по дереву |
| **Range scan** | Хорошо: данные отсортированы в SSTable | Хорошо: данные отсортированы в листьях |
| **Write amplification** | Есть: compaction переписывает данные | Есть: split страниц, WAL |
| **Сжатие** | Лучше: compaction создаёт компактные файлы | Хуже: фрагментация страниц |
| **Предсказуемость** | Хуже: compaction может вызвать всплески латентности | Лучше: стабильная производительность |

Ни один подход не лучше другого абсолютно. LSM-tree выигрывает на write-heavy нагрузках,
B-tree — на read-heavy с частыми point lookup.

## Почему LSM стал стандартом

В середине 2000-х объёмы данных начали расти взрывными темпами: социальные сети, мобильные приложения,
IoT, event streaming. Рабочие нагрузки сместились в сторону write-heavy, и B-tree начал
упираться в ограничения random I/O.

LSM-tree оказался идеальным ответом на эту эволюцию:

**Последовательный ввод-вывод.** Все записи на критическом пути — последовательные. Это максимально эффективный паттерн и для HDD (минимум перемещений головки), и для SSD (меньше усиление записи на уровне контроллера, дольше живёт диск).

**Высокий write throughput.** Запись в memtable — это операция в памяти. Сброс на диск — один последовательный write. LSM-движки обрабатывают в 5-10 раз больше записей в секунду, чем B-tree на том же железе.

**Эффективное сжатие.** Compaction создаёт плотные, хорошо сжимаемые файлы. B-tree страдает от фрагментации страниц — часть места пустует.

**Tiered storage.** LSM-tree естественным образом ложится на многоуровневое хранение: горячие данные в memtable (RAM), тёплые — на SSD, холодные — на S3. SlateDB, например, использует объектное хранилище как основной backend.

**Иммутабельность.** SSTable никогда не модифицируется после создания. Это упрощает concurrency, snapshot isolation и репликацию.

## Кто использует LSM-tree

LSM-tree — это не нишевая технология, а основа для огромной части современной инфраструктуры:

**Embedded storage engines:** LevelDB (Google), RocksDB (Meta), Pebble (Cockroach Labs), BadgerDB (Go-native).

**Распределённые SQL-базы:** CockroachDB (Pebble), TiDB/TiKV (RocksDB), YugabyteDB (DocDB на основе RocksDB).

**NoSQL-базы:** Apache Cassandra, Apache HBase, ScyllaDB.

**Потоковая обработка:** Apache Kafka Streams и Apache Flink используют RocksDB как state backend.

**Поиск:** Elasticsearch/Lucene используют LSM-подобные структуры для инвертированных индексов.

**Time-series:** InfluxDB построен на LSM.

RocksDB от Meta заслуживает отдельного упоминания — это, вероятно, самый широко используемый storage engine в мире. Он работает как embedded движок внутри десятков других баз данных и систем.

## Плюсы LSM-tree

- **Высокая пропускная способность на запись.** Последовательный ввод-вывод, буферизация в памяти, групповой сброс на диск.
- **Эффективное использование диска.** Compaction убирает мёртвые данные, файлы хорошо сжимаются.
- **Range scan.** Данные отсортированы и в memtable, и в SSTable.
- **Crash recovery.** WAL гарантирует durability. Потеря — максимум данные между последним fsync и крашем.
- **Неизменяемость SSTable.** SSTable никогда не модифицируется после создания. Это упрощает параллельный доступ, изоляцию снимков состояния и репликацию.
- **Масштабируемость.** Естественное разделение на уровни, совместимость с tiered storage.

## Минусы LSM-tree

- **Read amplification.** Чтение может потребовать проверки нескольких SSTable. Bloom filter помогает, но не устраняет проблему полностью.
- **Усиление записи из-за compaction.** Данные переписываются на каждом уровне compaction. При уровневой стратегии compaction объём реально записанных на диск данных может превышать объём пользовательских записей в 10-30 раз.
- **Всплески латентности.** Compaction конкурирует с пользовательскими операциями за CPU и I/O. Write stall — известная проблема, когда memtable заполняется быстрее, чем compaction успевает обработать SSTable.
- **Сложность настройки.** У RocksDB более 100 конфигурируемых параметров. Выбор стратегии compaction, размеров memtable, уровней — нетривиальная задача.
- **Space amplification.** До завершения compaction мёртвые данные занимают место на диске.

## Что дальше

Теперь, когда мы понимаем общую картину, будем реализовывать каждый компонент по отдельности:

1. **Skip List** — структура данных для memtable, поддерживающая O(log n) операции и упорядоченный обход
2. **SSTable** — формат файла с data blocks, index block, bloom filter и footer
3. **WAL** — доработаем наш append-only log для роли write-ahead log
4. **LSM Engine** — соберём всё вместе: путь записи, путь чтения, flush

Начнём со skip list в следующей главе.
