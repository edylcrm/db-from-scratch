В прошлой главе мы написали skip list — отсортированную структуру данных в памяти, которая служит memtable в LSM-tree.
Но «в памяти» — это и плюс, и проблема. Если процесс упал, содержимое памяти исчезает. Все записи, которые попали в
memtable, но ещё не были сброшены на диск в SSTable — потеряны.

WAL (Write-Ahead Log) решает эту проблему.

## Что такое WAL

WAL — это простой принцип: **прежде чем изменить данные в памяти, запиши намерение на диск**. Если процесс упал — при
следующем старте мы прочитаем WAL и восстановим всё, что было в memtable.

Это тот же append-only log, который мы написали в главе 1.1. Но теперь у него другая роль: WAL не является основным
хранилищем данных. Он существует только ради durability — чтобы данные пережили аварийное завершение процесса.
Как только memtable сбрасывается на диск в виде SSTable, соответствующий WAL можно удалить.

Название «Write-Ahead» буквально означает «запись заранее» — мы пишем в лог **до** того, как данные попадают в основное хранилище (memtable).

## Как WAL встраивается в путь записи

Без WAL путь записи выглядит так:

```
  Put("user:42", "alice")

  ┌──────────────────────┐
  │  SkipList (память)   │
  │  user:42 → alice     │  ← единственная копия данных
  └──────────────────────┘

  Процесс упал → данные потеряны.
```

С WAL добавляется один шаг перед записью в memtable:

```
  Put("user:42", "alice")

  Шаг 1: запись в WAL (диск)
  ┌───────────────────────────────────────────┐
  │  WAL файл (append-only)                   │
  │  [...] [PUT user:42 → alice]              │
  │                             ▲             │
  │                    sequential write       │
  └───────────────────────────────────────────┘
          │
          │  только после успешной записи на диск
          ▼
  Шаг 2: запись в memtable (память)
  ┌───────────────────────────────────────────┐
  │  SkipList                                 │
  │  user:42 → alice                          │
  └───────────────────────────────────────────┘

  Процесс упал → восстанавливаем memtable из WAL.
```

Порядок критически важен: **сначала WAL, потом memtable**. Если процесс упал между шагами 1 и 2 — данные есть в WAL
и будут восстановлены. Если упал до шага 1 — клиент не получил подтверждения, значит запись не произошла.
Промежуточного состояния (данные в памяти, но не на диске) больше нет.

## Формат записи

Каждая запись в WAL имеет формат:

```
  ┌──────┬───────┬──────┬──────────────────────────────────┐
  │ CRC  │ Len   │ Type │ Payload                          │
  │ 4B   │ 4B    │ 1B   │ N bytes                          │
  └──────┴───────┴──────┴──────────────────────────────────┘
```

Отличие от нашего append-only log из главы 1.1: появилось поле **Type**, которое указывает тип операции.
Это позволяет WAL хранить не только Put, но и Delete:

```
  Type = 0x01 (Put):
    Payload: [key_len (4B)] [val_len (4B)] [key] [value]

  Type = 0x02 (Delete):
    Payload: [key_len (4B)] [key]
```

CRC вычисляется от `Type + Payload`, Len — это размер `Type + Payload`.

## LSN: нумерация записей

Каждой записи в WAL присваивается уникальный монотонно возрастающий номер — **LSN** (Log Sequence Number).
LSN нужен для одной задачи: определить, какую часть WAL можно безопасно удалить.

```
  WAL файл:

  LSN=1        LSN=2        LSN=3        LSN=4
  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐
  │PUT a→1   │ │PUT b→2   │ │DEL a     │ │PUT c→3   │
  └──────────┘ └──────────┘ └──────────┘ └──────────┘

  MemTable сброшена в SSTable (flush), LSN на момент flush = 3.
  Записи с LSN <= 3 теперь на диске в SSTable → можно удалить.
  Запись LSN=4 ещё только в memtable → нужна в WAL.
```

## Восстановление после краша

При старте процесса мы проигрываем WAL от начала до конца, применяя каждую запись к пустому skip list:

```
  Старт процесса → memtable пустая

  Читаем WAL:
    LSN=1: PUT a→1   → skiplist.Set("a", "1")
    LSN=2: PUT b→2   → skiplist.Set("b", "2")
    LSN=3: DEL a     → skiplist.Delete("a")
    LSN=4: PUT c→3   → skiplist.Set("c", "3")
    EOF

  Memtable восстановлена: { b→2, c→3 }
  (ключ "a" удалён — Delete перекрыл Put)
```

Если последняя запись в WAL оборвана (краш на полузаписи) — CRC не сойдётся и мы её пропустим. Все предыдущие записи целы.

## Жизненный цикл WAL

WAL существует ровно столько, сколько живёт соответствующая memtable:

```
  1. Создаём новую memtable + новый WAL файл
     ┌─────────────┐    ┌──────────────┐
     │ wal-001.log │    │ memtable #1  │
     └─────────────┘    └──────────────┘

  2. Принимаем записи: WAL → memtable
     ┌─────────────┐    ┌──────────────┐
     │ PUT a→1     │    │ a→1          │
     │ PUT b→2     │    │ b→2          │
     │ PUT c→3     │    │ c→3          │
     └─────────────┘    └──────────────┘

  3. Memtable заполнилась → flush в SSTable
     Создаём новую memtable + новый WAL
     ┌─────────────┐    ┌──────────────┐    ┌──────────┐
     │ wal-001.log │    │ memtable #1  │───►│ SST #1   │
     │ (старый)    │    │ (immutable)  │    │(на диске)│
     └─────────────┘    └──────────────┘    └──────────┘
     ┌─────────────┐    ┌──────────────┐
     │ wal-002.log │    │ memtable #2  │
     │ (новый)     │    │ (активная)   │
     └─────────────┘    └──────────────┘

  4. Flush завершён → удаляем старый WAL
     wal-001.log больше не нужен: данные теперь в SST #1
```

## Проблема: fsync на каждую запись

Записать данные в файл через `write()` недостаточно. Операционная система буферизирует записи в page cache и 
сбрасывает их на физический диск в удобный для себя момент. Если сервер потерял питание между `write()` 
и реальной записью на диск — данные потеряны, хотя `write()` вернул успех.

Чтобы гарантировать, что данные действительно на диске, нужен **fsync** — системный вызов, который заставляет ОС 
сбросить все буферы на физический носитель.

```
  write() → данные в page cache ОС (в RAM)      ← быстро (~1 мкс)
  fsync() → данные на физическом диске           ← медленно (~0.1-10 мс)

  Без fsync:
    write("PUT a→1") → ОС: "ок, записал" (на самом деле в RAM)
    CRASH → данные потеряны

  С fsync:
    write("PUT a→1") → ОС: "ок, записал" (в RAM)
    fsync()          → ОС: "ок, на диске" (гарантированно)
    CRASH → данные на диске, восстановимы
```

В Go fsync — это `file.Sync()`.

Проблема в том, что fsync медленный. На HDD это 1-10 мс (диск должен физически повернуться). 
На NVMe SSD — порядка 0.05-0.2 мс. Если мы делаем fsync на каждую запись и хотим принимать 100 000 записей в секунду, 
каждый fsync должен занимать не более 10 мкс. Даже на NVMe мы не укладываемся.

```
  fsync на каждую запись:

  Put #1 ──► write ──► fsync (0.1ms) ──► ответ клиенту
  Put #2 ──► write ──► fsync (0.1ms) ──► ответ клиенту
  Put #3 ──► write ──► fsync (0.1ms) ──► ответ клиенту

  Максимум: ~10 000 записей/сек на NVMe
  На HDD: ~100-1000 записей/сек
```

Это фундаментальное ограничение. Каждый fsync — это обращение к контроллеру диска с командой «сбрось всё на носитель».
Обойти физику нельзя, но можно сократить количество fsync.

## Решение: Group Commit

Идея group commit: вместо fsync на каждую запись, мы **собираем пачку записей за короткий интервал** (например, 2 мс)
и делаем **один fsync на всю пачку**. Все записи в пачке получают гарантию durability одновременно.

```
  Group commit (интервал 2ms):

  0.0ms: Put #1 ──► write в буфер ──► ожидание...
  0.3ms: Put #2 ──► write в буфер ──► ожидание...
  0.7ms: Put #3 ──► write в буфер ──► ожидание...
  1.1ms: Put #4 ──► write в буфер ──► ожидание...
  1.8ms: Put #5 ──► write в буфер ──► ожидание...
  2.0ms: ──────────── fsync ───────────────────────
         Put #1 ◄── ответ
         Put #2 ◄── ответ
         Put #3 ◄── ответ
         Put #4 ◄── ответ
         Put #5 ◄── ответ

  1 fsync вместо 5. Время ответа каждого Put выросла
  (максимум +2ms), но пропускная способность — в разы выше.
```

Компромисс очевиден: мы жертвуем производительностью отдельной операции (в худшем случае +2 мс ожидания до следующего fsync) 
ради значительного роста пропускной способности. Если за 2 мс пришло 100 записей — мы сэкономили 99 fsync.

### Как устроен group commit

Архитектура строится на отдельной горутине, которая собирает записи из канала и периодически делает flush + fsync:

```
  Горутины-писатели                   commitLoop (отдельная горутина)
  ┌──────────┐                        ┌──────────────────────────────┐
  │ Put #1   │──┐                     │                              │
  └──────────┘  │                     │  1. Жду записи из канала     │
  ┌──────────┐  │   pending channel   │  2. Собираю пачку за ~2ms    │
  │ Put #2   │──┼────────────────────►│  3. Пишу всё в буфер         │
  └──────────┘  │                     │  4. flush + fsync            │
  ┌──────────┐  │                     │  5. Уведомляю всех через     │
  │ Put #3   │──┘                     │     done channel             │
  └──────────┘                        └──────────────────────────────┘
       ▲                                          │
       │              done channel                │
       └──────────────────────────────────────────┘
```

Каждый вызов `WritePut`:
1. Формирует бинарную запись
2. Отправляет её в `pending` channel
3. Блокируется, ожидая ответ в своём `done` channel

`commitLoop` в отдельной горутине:
1. Ждёт первую запись из `pending`
2. Собирает все записи, которые накопились за ~2 мс
3. Записывает всю пачку в буфер, делает один `Flush()` + `Sync()`
4. Отправляет результат (nil или ошибку) в `done` channel каждого писателя

## Что используется в реальных БД

WAL — это не факультативный компонент, а обязательная часть практически любой базы данных.

**PostgreSQL** использует WAL как основу своей durability-модели. Вместо того чтобы при каждом коммите
записывать все изменённые страницы данных на диск, PostgreSQL записывает только WAL — это значительно дешевле,
потому что WAL пишется последовательно. PostgreSQL поддерживает group commit через параметр `commit_delay` — задержку в
микросекундах перед fsync, которая позволяет нескольким транзакциям попасть в один fsync.

**RocksDB** реализует group commit: когда несколько потоков одновременно пишут в одну БД, все ожидающие записи 
объединяются и записываются в WAL за один раз, с одним fsync. При этом RocksDB не добавляет искусственных
задержек — группировка происходит естественным образом за счёт конкуренции потоков. Максимальный размер группы — 1 МБ.

**Kafka** — по сути целиком построена вокруг append-only log. Каждый partition — это WAL, в который продюсеры дописывают сообщения.

**etcd** — хранилище состояния Kubernetes — логирует каждую мутацию через WAL перед применением к основному хранилищу.

## Плюсы WAL

- **Durability.** Гарантия того, что подтверждённые данные переживут краш. Потеря — максимум данные после последнего fsync (при group commit — до 2 мс записей).
- **Последовательная запись.** WAL всегда пишется в конец файла — самый эффективный паттерн для любого типа диска.
- **Простое восстановление.** При старте просто проигрываем лог от начала до конца. Не нужны сложные механизмы восстановления из битых структур.
- **Декомпозиция.** WAL отделяет durability от основного хранилища. Memtable может быть любой in-memory структурой — WAL обеспечивает сохранность.

## Минусы WAL

- **Двойная запись.** Каждая операция записывается дважды: в WAL и в memtable (а потом ещё в SSTable). Это неизбежная цена за durability.
- **fsync — узкое место.** Даже на NVMe fsync занимает сотни микросекунд. Без group commit это ограничивает пропускную способность.
- **Восстановление O(n).** Чем больше WAL, тем дольше старт. Поэтому важно своевременно удалять WAL после flush memtable в SSTable.
- **Дополнительное место на диске.** WAL дублирует данные, которые есть в memtable. Обычно это небольшой объём (размер memtable — единицы-десятки МБ).

### GroupCommitWAL (fsync раз в ~2 мс на пачку записей)

Важно: `GroupCommitWAL` должен реализовывать тот же интерфейс, что и обычный `WAL`, чтобы остальной код 
(LSM engine, memtable) не знал, какая реализация используется. Для этого выделим общий интерфейс:

```go
// WALWriter — общий интерфейс для WAL и GroupCommitWAL.
// LSM engine работает с этим интерфейсом и не знает,
// делается ли fsync на каждую запись или батчами.
type WALWriter interface {
    WritePut(key, value []byte) (LSN, error)
    WriteDelete(key []byte) (LSN, error)
    Recover(memTable *SkipList) error
    Close() error
}
```

Это позволяет переключаться между реализациями одной строкой конфигурации — без изменения кода engine.

Обратите внимание:

- **`bufio.Writer`** с буфером 64 КБ — записи накапливаются в памяти и сбрасываются на диск одним вызовом. Это уменьшает количество системных вызовов `write()`.
- **CRC считается от `Type + Payload`**, а не от всей записи — Len и CRC не входят в покрытие контрольной суммы.
- **`Recover` игнорирует оборванные записи** — в нашей учебной реализации мы просто пропускаем запись с невалидным CRC и прекращаем чтение. Это **не штатная ситуация**: в production-решениях оборванная запись означает, что WAL повреждён и часть данных потенциально потеряна. Реальные БД решают это через блоки фиксированного размера (как в LevelDB/RocksDB — мы разбирали это в главе 1.1), через реплики (данные есть на другой ноде) или через checksumming на уровне всего WAL-файла. В нашем случае мы упрощаем: если CRC не сошёлся — считаем, что это был краш на полузаписи, и останавливаемся.
- **`done chan error`** в GroupCommitWAL — каждый писатель создаёт свой канал и блокируется на нём, пока commitLoop не сделает fsync и не отправит результат.

func TestWALGroupCommitCorrectness(t *testing.T) {
	// Проверяем что group commit не теряет данные при конкурентной записи.
	walPath := filepath.Join(t.TempDir(), "wal-gc")
	wal, err := NewGroupCommitWAL(walPath)
	if err != nil {
		t.Fatal(err)
	}

	const (
		goroutines = 20
		writesEach = 500
	)

	var wg sync.WaitGroup
	var mu sync.Mutex
	written := make(map[string]string)

	for g := 0; g < goroutines; g++ {
		wg.Add(1)
		go func(id int) {
			defer wg.Done()
			for i := 0; i < writesEach; i++ {
				key := fmt.Sprintf("g%d-k%d", id, i)
				val := fmt.Sprintf("g%d-v%d", id, i)
				if _, err := wal.WritePut([]byte(key), []byte(val)); err != nil {
					t.Errorf("WritePut: %v", err)
					return
				}
				mu.Lock()
				written[key] = val
				mu.Unlock()
			}
		}(g)
	}
	wg.Wait()
	wal.file.Close()

	// Восстанавливаем и проверяем что все записи на месте:
	wal2, _ := OpenWAL(walPath)
	mem := NewSkipList()
	if err := wal2.Recover(mem); err != nil {
		t.Fatal(err)
	}

	for key, wantVal := range written {
		gotVal, ok := mem.Get([]byte(key))
		if !ok || gotVal == nil {
			t.Errorf("key %q missing after recovery", key)
			continue
		}
		if string(gotVal) != wantVal {
			t.Errorf("key %q: got %q, want %q", key, gotVal, wantVal)
		}
	}
	t.Logf("recovered %d/%d writes correctly", len(written), goroutines*writesEach)
}

func TestWALTruncate(t *testing.T) {
	// Проверяем TruncateBefore: после truncate старые записи недоступны,
	// новые — на месте.
	dir := t.TempDir()
	walPath := filepath.Join(dir, "wal")

	wal, _ := OpenWAL(walPath)

	// Пишем 10 записей и запоминаем LSN после 5-й:
	var lsnAfter5 LSN
	for i := 0; i < 10; i++ {
		key := fmt.Appendf(nil, "key-%d", i)
		lsn, err := wal.WritePut(key, []byte("val"))
		if err != nil {
			t.Fatal(err)
		}
		if i == 4 {
			lsnAfter5 = lsn
		}
	}

	// Truncate всё до lsnAfter5:
	if err := wal.TruncateBefore(lsnAfter5 + 1); err != nil {
		t.Fatal(err)
	}
	wal.file.Close()

	// Восстанавливаем — должны увидеть только записи с LSN > lsnAfter5:
	wal2, _ := OpenWAL(walPath)
	mem := NewSkipList()
	wal2.Recover(mem)

	// Записи 0-4 должны отсутствовать:
	for i := 0; i < 5; i++ {
		key := fmt.Appendf(nil, "key-%d", i)
		if _, ok := mem.Get(key); ok {
			t.Errorf("key-%d should have been truncated", i)
		}
	}

	// Записи 5-9 должны быть:
	for i := 5; i < 10; i++ {
		key := fmt.Appendf(nil, "key-%d", i)
		if _, ok := mem.Get(key); !ok {
			t.Errorf("key-%d missing after truncate", i)
		}
	}
}
