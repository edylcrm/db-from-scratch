Начнём с самого примитивного подхода к хранению данных, который только можно придумать. Один файл, в который
мы дописываем записи одну за другой. Никаких индексов, никаких деревьев — просто файл.

Звучит бесполезно? На самом деле именно этот принцип лежит в основе Bitcask — storage engine,
который использовался в распределённой базе данных Riak. И именно от append-only log мы будем отталкиваться,
чтобы постепенно прийти к полноценному LSM-движку.

## Формат записи

Каждая запись в нашем файле будет иметь фиксированную структуру:

```
┌──────────┬─────────┬─────────┬─────────────┬─────────────┐
│  CRC32   │ key_len │ val_len │     key     │    value    │
│ (4 bytes)│(4 bytes)│(4 bytes)│ (key_len)   │ (val_len)   │
└──────────┴─────────┴─────────┴─────────────┴─────────────┘
```

Первые 12 байт — заголовок фиксированного размера. За ним идут данные переменной длины: ключ и значение.
Такой формат позволяет последовательно читать записи из файла: прочитали 12 байт заголовка, узнали длины ключа и 
значения, прочитали ровно столько данных, перешли к следующей записи.

Наше хранилище будет предоставлять три операции:

- **Set(key, value)** — записать пару ключ-значение
- **Get(key)** — найти значение по ключу
- **Delete(key)** — удалить ключ

Delete — это тоже запись. Мы не можем удалить что-то из середины файла,
поэтому удаление — это запись специального маркера, так называемого **tombstone**.
При чтении, встретив tombstone, мы понимаем, что ключ был удалён.

## CRC32: проверка целостности

Первые 4 байта каждой записи — это контрольная сумма CRC32 (Cyclic Redundancy Check, 32-bit). CRC32 — это хеш-функция,
которая принимает произвольные данные на вход и возвращает 32-битное число. Она не криптографическая, но быстрая и 
хорошо ловит случайные повреждения: битовые ошибки на диске, обрыв записи при аварии, повреждение при передаче по сети.

Идея простая: при записи мы вычисляем CRC32 от `key_len + val_len + key + value` и сохраняем его в начале записи.
При чтении — вычисляем CRC32 заново и сравниваем с сохранённым. Если не совпадает — запись повреждена.

В Go CRC32 доступен из стандартной библиотеки:

```go
import "hash/crc32"

checksum := crc32.ChecksumIEEE(data)
```

CRC32 используется повсеместно: в форматах файлов (PNG, ZIP), сетевых протоколах (Ethernet), а также
в базах данных — например, в RocksDB и Pebble для проверки целостности блоков в SSTable.

## Что делать с битыми записями

CRC32 позволяет обнаружить повреждённую запись — но что с ней делать дальше? Вопрос не такой простой, как кажется.
Append-only log — это последовательность записей, и чтобы прочитать запись N, нужно сначала прочитать заголовки
записей 1..N-1. Если запись посередине повреждена и `key_len` содержит мусор — мы не знаем, где начинается
следующая запись, и теряем весь «хвост» файла.

Существует несколько стратегий решения этой проблемы.

**Truncate на битой записи.** Самый простой подход. Встретили невалидный CRC32 — считаем, что всё после этой точки
повреждено, и обрезаем файл.

**Magic bytes + сканирование.** В начало каждой записи добавляется фиксированная последовательность байтов (например, `0xDEADBEEF`).
При обнаружении битой записи мы сканируем файл побайтово вперёд, ищем следующее вхождение magic bytes и пытаемся читать оттуда.
Это позволяет восстановить данные после повреждённого участка, но работает медленно и есть риск ложных 
срабатываний — magic bytes могут случайно встретиться в пользовательских данных.

**Блоки фиксированного размера.** Файл делится на блоки фиксированного размера (например, 32 КБ).
Записи укладываются в блоки, и если запись не помещается целиком — она разбивается на фрагменты с типом `FULL`, `FIRST`, `MIDDLE`, `LAST`.
Каждый фрагмент имеет свой CRC. Если блок повреждён — мы теряем только его содержимое и переходим к следующему блоку
по фиксированному смещению. Это самый надёжный подход: повреждение одного блока не влияет на остальные.

> **В нашей реализации** мы не будем обрабатывать битые записи — это учебный проект, и на данном этапе важнее
разобраться с основной механикой append-only log. Но знать о существовании этих стратегий полезно: когда мы дойдём
до реализации WAL, подход с блоками фиксированного размера нам пригодится.

## Бинарная сериализация: как записать числа в байты

Длины ключа и значения — это числа типа `uint32`. Но файл — это последовательность байтов. Нам нужен способ
превратить число в байты и обратно. В Go есть несколько подходов.

### Подход 1: `encoding/binary` — ручная запись в буфер

Это то, что мы будем использовать. Пакет `encoding/binary` позволяет записать число в слайс байтов с явным указанием порядка байтов:

```go
header := make([]byte, 12)
binary.LittleEndian.PutUint32(header[0:4], checksum)
binary.LittleEndian.PutUint32(header[4:8], uint32(len(key)))
binary.LittleEndian.PutUint32(header[8:12], uint32(len(value)))
```

Максимальный контроль, никаких аллокаций, всё прозрачно. Для storage engine это идеальный
вариант — мы точно знаем, что, куда и в каком порядке пишем.

### Подход 2: `binary.Write` — запись через io.Writer

```go
buf := new(bytes.Buffer)
binary.Write(buf, binary.LittleEndian, uint32(len(key)))
```

Удобнее для сериализации структур целиком, но внутри использует рефлексию и аллоцирует память.
Для hot path в storage engine — лишний overhead.

### Подход 3: `encoding/gob`

```go
encoder := gob.NewEncoder(buf)
encoder.Encode(myStruct)
```

Gob — Go-специфичный формат сериализации. Удобен для RPC между Go-сервисами, но для хранения на диске не подходит: 
формат привязан к Go, нет фиксированного бинарного layout, и размер выходных данных непредсказуем. Мы же хотим точно контролировать каждый байт.

## Little Endian vs Big Endian

В коде выше мы используем `binary.LittleEndian`. Что это значит и почему именно little endian?

Когда мы записываем многобайтовое число (например, `uint32`) в память или в файл, возникает вопрос: в каком порядке располагать байты?

Возьмём число `0x01020304` (16909060 в десятичной). Оно состоит из четырёх байтов: `01`, `02`, `03`, `04`.

**Big Endian** — старший байт первым:

```
Адрес:  0    1    2    3
Байты: [01] [02] [03] [04]
```

Это «человеческий» порядок — как мы привыкли читать числа слева направо.

**Little Endian** — младший байт первым:

```
Адрес:  0    1    2    3
Байты: [04] [03] [02] [01]
```

Выглядит «задом наперёд», но именно так хранят числа процессоры x86 и x86-64 (Intel, AMD), а также большинство реализаций ARM и RISC-V.

### Где что используется

**Little Endian:** процессоры x86/x86-64 (практически все серверы и десктопы), ARM (в большинстве конфигураций), RISC-V.
На уровне файловых форматов — многие БД, включая RocksDB/Pebble, протокол SMB.

**Big Endian:** сетевые протоколы (TCP/IP, HTTP). В сетевом мире big endian настолько стандартен, что его называют
**network byte order**. Также big endian используют старые архитектуры (Motorola 68k, IBM z/Series), 
Java (внутренне всегда big endian) и многие бинарные форматы файлов.

### Что выбрать для storage engine

Мы пишем storage engine, который будет работать на x86/ARM. Little endian — нативный порядок для этих архитектур.
Запись и чтение в нативном порядке не требует конвертации — процессор может работать с данными напрямую.
Это не критично для нашего учебного проекта, но в production-реализациях это имеет значение.

## Сложность операций

Давайте оценим сложность операций нашего append-only log:

| Операция | Сложность | Почему |
|----------|-----------|--------|
| Set      | O(1)      | Дописываем запись в конец файла — один sequential write |
| Get      | O(n)      | Нужно просканировать файл от конца к началу, ищем последнюю запись с нужным ключом |
| Delete   | O(1)      | Это просто Set с tombstone-маркером |

Запись быстрая — это сильная сторона подхода. Чтение O(n) — это проблема.

## Плюсы

- **Простота реализации.** Весь storage engine — это один файл и пара десятков строк кода. Легко понять, легко отладить.
- **Быстрая запись.** Sequential write в конец файла — самый эффективный паттерн для дисков.
- **Crash safety.** Если процесс упал посередине записи, мы потеряем максимум одну последнюю запись. CRC32 позволит обнаружить неполную запись при восстановлении. Все предыдущие записи останутся целыми.
- **Иммутабельность.** Мы никогда не модифицируем существующие данные — только дописываем. Это упрощает concurrency и восстановление после сбоев.

## Минусы

- **Медленное чтение.** O(n) на каждый Get — неприемлемо для любого реального применения.
- **Файл растёт бесконечно.** Даже если ключ обновляется тысячу раз, все тысяча записей остаются в файле. Нужен механизм compaction, чтобы очищать устаревшие данные.
- **Нет range scan.** Невозможно эффективно получить «все ключи от A до Z» — данные не отсортированы.
- **Все ключи в файле.** При рестарте нужно просканировать весь файл, чтобы восстановить актуальное состояние.

func TestCrashRecovery(t *testing.T) {
	dir := t.TempDir()
	path := filepath.Join(dir, "crash.log")

	// Шаг 1: пишем несколько валидных записей и закрываем базу.
	db, err := Open(path)
	if err != nil {
		t.Fatal(err)
	}
	db.Set([]byte("key1"), []byte("value1"))
	db.Set([]byte("key2"), []byte("value2"))
	db.Set([]byte("key3"), []byte("value3"))
	db.Close()

	// Шаг 2: симулируем краш — дописываем в конец файла мусор разной длины.
	// Это имитирует ситуацию когда процесс упал в середине записи:
	// может быть записан заголовок без данных, или данные без CRC, или вообще случайные байты.
	f, err := os.OpenFile(path, os.O_APPEND|os.O_WRONLY, 0644)
	if err != nil {
		t.Fatal(err)
	}
	// Случай 1: оборванный заголовок (меньше 12 байт)
	f.Write([]byte{0xDE, 0xAD, 0xBE}) // только 3 байта вместо 12
	f.Close()

	// Шаг 3: открываем заново — должны увидеть все три валидных записи.
	db2, err := Open(path)
	if err != nil {
		t.Fatal(err)
	}
	defer db2.Close()

	val, err := db2.Get([]byte("key1"))
	if err != nil {
		t.Fatalf("key1 missing after crash recovery: %v", err)
	}
	if string(val) != "value1" {
		t.Fatalf("key1: got %q, want %q", val, "value1")
	}

	val, err = db2.Get([]byte("key2"))
	if err != nil {
		t.Fatalf("key2 missing after crash recovery: %v", err)
	}
	if string(val) != "value2" {
		t.Fatalf("key2: got %q, want %q", val, "value2")
	}

	val, err = db2.Get([]byte("key3"))
	if err != nil {
		t.Fatalf("key3 missing after crash recovery: %v", err)
	}
	if string(val) != "value3" {
		t.Fatalf("key3: got %q, want %q", val, "value3")
	}
}

	db2, _ := Open(path)
	defer db2.Close()

	val, err := db2.Get([]byte("before"))
	if err != nil {
		t.Fatalf("valid record lost after crash: %v", err)
	}
	if string(val) != "crash" {
		t.Fatalf("got %q, want %q", val, "crash")
	}
}

	db := openTemp(b)

	const n = 10_000
	for i := 0; i < n; i++ {
		key := fmt.Appendf(nil, "key-%06d", i)
		db.Set(key, []byte("value"))
	}

	// Benchmark Get — должен расти линейно с n:
	b.ResetTimer()
	for i := 0; i < b.N; i++ {
		key := fmt.Appendf(nil, "key-%06d", i%n)
		db.Get(key)
	}
}

// openTemp принимает testing.TB — работает и для тестов и для бенчмарков.
func openTemp(tb testing.TB) *Log {
	tb.Helper()
	path := filepath.Join(tb.TempDir(), "test.log")
	db, err := Open(path)
	if err != nil {
		tb.Fatal(err)
	}
	tb.Cleanup(func() { db.Close() })
	return db
}
