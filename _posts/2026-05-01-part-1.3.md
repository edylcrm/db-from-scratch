Мы прошли первые два шага на пути к storage engine. Давайте оглянемся: что мы построили, чего добились, с чем столкнулись и куда двигаемся дальше.

## Что мы построили

Наше хранилище состоит из двух слоёв:

```
  ┌────────────────────────────────────────────────┐
  │               HashDB (глава 1.2)               │
  │                                                │
  │  ┌──────────────────────────────────────────┐  │
  │  │  In-Memory Index                         │  │
  │  │  map[string]valueLocation                │  │
  │  │                                          │  │
  │  │  "user:1"  → {offset: 0,    size: 35}    │  │
  │  │  "user:2"  → {offset: 512,  size: 41}    │  │
  │  │  "config"  → {offset: 1024, size: 28}    │  │
  │  └──────────────────┬───────────────────────┘  │
  │                     │                          │
  └─────────────────────┼──────────────────────────┘
                        │ ReadAt(buf, offset)
                        ▼
  ┌────────────────────────────────────────────────┐
  │               Log (глава 1.1)                  │
  │                                                │
  │  Append-only файл на диске                     │
  │  ┌──────┐┌──────┐┌──────┐┌──────┐┌──────┐      │
  │  │ rec0 ││ rec1 ││ rec2 ││ rec3 ││ ...  │      │
  │  └──────┘└──────┘└──────┘└──────┘└──────┘      │
  │                                                │
  │  Каждая запись:                                │
  │  [CRC32 | key_len | val_len | key | value]     │
  └────────────────────────────────────────────────┘
```

**Log** (глава 1.1) — append-only файл с бинарным форматом записей. CRC32 для проверки целостности, Little Endian для числовых полей, tombstone для удалений. Все операции дописываются в конец файла.

**HashDB** (глава 1.2) — in-memory hash table поверх Log. Хранит маппинг `ключ → {offset, size}` для мгновенного доступа к данным на диске. При старте восстанавливает индекс полным сканом файла.

## Как этим пользоваться

API получился минимальным и понятным:

```go
// Открываем (или создаём) хранилище
db, err := storage.OpenHashDB("data/my.db")
defer db.Close()

// Записываем
db.Set([]byte("user:1"), []byte(`{"name":"alice","age":30}`))
db.Set([]byte("user:2"), []byte(`{"name":"bob","age":25}`))

// Читаем
val, err := db.Get([]byte("user:1"))
// val = {"name":"alice","age":30}

// Удаляем
db.Delete([]byte("user:2"))

// Статистика
stats, _ := db.Stats()
// stats.LiveKeys = 1
// stats.SpaceAmplification = TotalFileBytes / LiveDataBytes
```

Хранилище потокобезопасно: несколько горутин могут одновременно читать, запись блокирует всех. Данные переживают перезапуск процесса — при открытии индекс восстанавливается из файла.

## Где этот подход нашёл применение

Наша реализация — это упрощённый **Bitcask**, storage engine, который был создан для распределённой базы данных Riak. Bitcask появился в 2010 году, когда инженеры Basho Technologies искали хранилище, которое обеспечивало бы предсказуемую латентность и высокий write throughput для каждой ноды кластера Riak.

Идея оказалась настолько удачной для определённого класса задач, что принцип «append-only log + in-memory hash index» используется и за пределами Bitcask:

- **Riak KV** — использовал Bitcask как дефолтный storage backend. Каждый vnode (виртуальная нода) кластера Riak работал со своим независимым экземпляром Bitcask. Basho Technologies обанкротилась в 2017 году, но проект продолжает жить как **OpenRiak** — форк, поддерживаемый сообществом при участии Erlang Ecosystem Foundation. Версия 3.4 находится в активной разработке, версия 3.6 планируется на 2026 год.
- **Go-реализации Bitcask** — существуют активно поддерживаемые open-source реализации на Go, например `go.mills.io/bitcask/v2` (последний релиз — октябрь 2025). Используется как embedded KV-хранилище и как backend для распределённого key-value store Bitraft.
- **Паттерн WAL** — принцип append-only записи с последующим восстановлением лежит в основе Write-Ahead Log практически во всех современных базах данных. WAL в PostgreSQL, RocksDB, CockroachDB — это тот же append-only log, только с другим назначением (durability для in-memory структур, а не основное хранилище).

## С какими проблемами мы столкнулись

За время реализации мы обнаружили несколько фундаментальных ограничений этого подхода.

### Все ключи должны поместиться в RAM

Это главное ограничение. Индекс занимает ~80 байт на ключ в Go. Для 100 миллионов ключей — это ~8 ГБ RAM только под индекс, без учёта самих данных. Данные могут лежать на диске терабайтами, но количество уникальных ключей ограничено объёмом оперативной памяти.

Bitcask решает это горизонтальным шардированием: данные разносятся по нодам, каждая нода держит только свою часть ключей. Но это решение на уровне архитектуры, а не storage engine.

### Нет range scan

Hash table не поддерживает упорядоченный обход. Запрос «дай мне все ключи с префиксом `user:`» или «все записи с `id` от 1000 до 2000» требует полного перебора всего индекса. Для SQL-запросов с `WHERE`, `ORDER BY`, `BETWEEN` это неприемлемо.

Это не баг реализации — это фундаментальное свойство hash table как структуры данных. Чтобы его решить, нужна другая структура: отсортированная, с эффективным последовательным обходом.

### Файл растёт бесконечно

Каждое обновление ключа оставляет старую запись мёртвой в файле. Каждое удаление добавляет tombstone, но не удаляет предыдущие записи. Файл только растёт.

```
  Файл после 1000 обновлений одного ключа:

  ┌───────┐┌───────┐┌───────┐       ┌───────┐
  │v1 мёрт││v2 мёрт││v3 мёрт│ . . . │v1000  │
  └───────┘└───────┘└───────┘       └───────┘
  ▲         ▲         ▲              ▲
  мусор     мусор     мусор          живая запись

  999 из 1000 записей — мусор.
  SpaceAmplification ≈ 1000x
```

В production-реализациях эту проблему решает compaction (merge) — процесс, который переписывает файл, оставляя только живые записи. В нашей реализации compaction нет.

### Медленный старт

При перезапуске процесса нужно просканировать весь файл, чтобы восстановить индекс. Для файла в несколько гигабайт это может занять секунды или даже минуты. Bitcask решает это hint-файлами, которые мы разобрали в главе 1.2.

## Плюсы и минусы

| | Плюс | Минус |
|---|---|---|
| **Запись** | O(1), sequential, максимально эффективный паттерн для дисков | Файл растёт бесконечно без compaction |
| **Чтение** | O(1) lookup + один random read с диска | Все ключи должны поместиться в RAM |
| **Range scan** | — | Невозможен: hash table не поддерживает упорядоченность |
| **Crash safety** | Потеря максимум одной последней записи, CRC для обнаружения повреждений | Восстановление индекса O(n) при старте |
| **Простота** | Реализация в ~200 строк Go, легко понять и отладить | Не подходит для сложных запросов |

## Используется ли это в 2026 году?

Bitcask как конкретный продукт — нишевый. Riak, для которого он был создан, после банкротства Basho в 2017 году перешёл в руки сообщества (OpenRiak) и используется в основном в legacy-инсталляциях, хотя активная разработка продолжается.

Но сам паттерн «append-only log + in-memory index» живёт и здравствует:

- **WAL** в каждой современной БД — это append-only log.
- **Kafka** — по сути гигантский append-only log с индексами.
- **Embedded KV-хранилища** — для задач с ограниченным количеством ключей и требованием предсказуемой латентности подход Bitcask остаётся актуальным.

Для новых проектов, которым нужен полноценный storage engine, индустрия перешла на **LSM-tree** (LevelDB, RocksDB, Pebble) — подход, который снимает ограничение «все ключи в RAM» и добавляет range scan. Именно LSM мы будем строить дальше.

## Что дальше

Мы определили две главные проблемы hash index: отсутствие range scan и требование держать все ключи в памяти. Обе решаются заменой hash table на отсортированную структуру данных.

В следующей главе мы разберём **skip list** — вероятностную структуру, которая даёт O(log n) на поиск и вставку, поддерживает упорядоченный обход и станет основой нашей memtable в LSM-движке.
