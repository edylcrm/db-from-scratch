В предыдущей главе мы написали append-only log. Запись работает за O(1),
но каждый Get — это полный скан файла от начала до конца.
При миллионе записей это уже неприемлемо.

Решение напрашивается: давайте держать в памяти индекс, который для каждого ключа знает,
где именно в файле лежит его последнее значение. Тогда вместо сканирования всего файла мы
делаем один lookup в hash table и один точечный read с диска.

## Идея

Поверх нашего append-only log мы добавляем in-memory hash table: `map[string]int64`, где ключ — строковый ключ записи,
а значение — смещение (offset) в файле, по которому лежит последняя запись для этого ключа.

```
                    In-Memory Hash Table
                   ┌─────────────────────┐
                   │  "name"  → 0        │
                   │  "age"   → 42       │
                   │  "city"  → 81       │
                   └────────┬────────────┘
                            │
         offset lookup      │
                            ▼
  ┌──────────────────────────────────────────────────────────┐
  │  Файл на диске (append-only)                             │
  │                                                          │
  │  offset=0        offset=42       offset=81               │
  │  ┌────────────┐  ┌────────────┐  ┌─────────────┐         │
  │  │CRC|4|5|    │  │CRC|3|2|    │  │CRC|4|6|     │         │
  │  │name|alice  │  │age|25      │  │city|moscow  │         │
  │  └────────────┘  └────────────┘  └─────────────┘         │
  └──────────────────────────────────────────────────────────┘
```

## Как работают операции

### Set(key, value)

1. Запоминаем текущую позицию в файле (это будет offset новой записи)
2. Дописываем запись в конец файла (как раньше)
3. Обновляем hash table: `index[key] = offset`

```
  Set("name", "bob"):

  1. offset = текущий размер файла (например, 120)

  2. Дописываем в файл:
     ┌──────────┬─────────┬─────────┬──────┬─────┐
     │  CRC32   │ key_len │ val_len │ name │ bob │
     │  4 bytes │    4    │    3    │      │     │
     └──────────┴─────────┴─────────┴──────┴─────┘
     ^
     offset = 120

  3. index["name"] = 120  (перезаписываем старый offset)
```

### Get(key)

1. Ищем ключ в hash table → получаем offset
2. Делаем Seek на нужную позицию в файле
3. Читаем одну запись (заголовок + данные)
4. Проверяем CRC, возвращаем value

```
  Get("name"):

  1. index["name"] → offset = 120

  2. Seek(120) в файле

  3. Читаем запись:
     ┌──────────┬─────────┬─────────┬──────┬─────┐
     │  CRC32   │    4    │    3    │ name │ bob │
     └──────────┴─────────┴─────────┴──────┴─────┘
     ^
     позиция 120

  4. Проверяем CRC ✓ → возвращаем "bob"
```

Никакого сканирования. Один lookup в hash table (O(1)) и один точечный read с диска.
Неважно, сколько записей в файле — время чтения не растёт.

### Delete(key)

1. Дописываем tombstone в файл (как раньше)
2. Удаляем ключ из hash table: `delete(index, key)`

После удаления из индекса ключ перестаёт быть «видимым», хотя физически все его записи (включая tombstone) остаются в файле.

## Аналогия с Bitcask

То, что мы строим — это упрощённый Bitcask. В Bitcask эта in-memory структура называется 
**KeyDir** — это термин из [оригинального paper](https://riak.com/assets/bitcask-intro.pdf), а не название конкретного
типа в коде. В разных реализациях она может называться `KeyDirectory`, `index` или просто `map` — суть одна и та же.
Разница с нашей реализацией в деталях:

| | Наша реализация | Bitcask |
|---|---|---|
| Индекс | `map[string]int64` (только offset) | KeyDir: `map[key]{file_id, offset, size, timestamp}` |
| Файлы | Один файл | Несколько сегментов: один активный + неизменяемые старые |
| Compaction | Нет | Merge: старые сегменты сливаются, мёртвые записи удаляются |
| Восстановление | Полный скан файла при старте | Hint-файлы: содержат только ключи и offsets, ускоряют восстановление |

Но ключевая идея идентична: **все ключи хранятся в оперативной памяти**, а на диске лежат только данные. 
Чтение — это один lookup в hash table + один disk read.

## Восстановление после рестарта

У in-memory индекса есть неизбежная особенность: при перезапуске процесса hash table пропадает.
Чтобы восстановить индекс, нужно просканировать весь файл от начала до конца и заново построить mapping `ключ → offset`.

```
  Процесс запускается → файл уже содержит данные:

  offset=0           offset=23          offset=42
  ┌───────────────┐  ┌───────────────┐  ┌───────────────┐
  │ name → alice  │  │ age  → 25     │  │ name → bob    │
  └───────────────┘  └───────────────┘  └───────────────┘

  Скан файла слева направо:

  Шаг 1: index["name"] = 0
  Шаг 2: index["age"]  = 23
  Шаг 3: index["name"] = 42   ← перезаписываем, это более новое значение

  Итог: index = { "name": 42, "age": 23 }
```

Это O(n) по количеству записей в файле и происходит при каждом старте. Для файла в несколько гигабайт это
может занять ощутимое время. В Bitcask эта проблема решается через **hint-файлы**.

### Hint-файлы в Bitcask

Проблема понятна: чтобы восстановить индекс, нужно прочитать весь data-файл — каждую запись со всеми ключами и значениями.
Если данных терабайт, старт может занять минуты.

Идея hint-файлов: при compaction (merge) рядом с каждым новым data-файлом создаётся компактный файл,
содержащий только метаданные — без значений.

Сравним форматы записей:

```
  Data-файл (полная запись):
  ┌───────┬───────────┬─────────┬─────────┬─────────┬───────────┐
  │ CRC32 │ timestamp │ key_len │ val_len │   key   │   value   │
  │ 4 B   │   4 B     │  4 B    │  4 B    │  N B    │   M B     │
  └───────┴───────────┴─────────┴─────────┴─────────┴───────────┘

  Hint-файл (только метаданные, без value):
  ┌───────────┬─────────┬─────────┬──────────┬─────────┐
  │ timestamp │ key_len │ val_len │ val_pos  │   key   │
  │   4 B     │  4 B    │  4 B    │   8 B    │  N B    │
  └───────────┴─────────┴─────────┴──────────┴─────────┘
```

Ключевое отличие: вместо самого value хранится `val_pos` — смещение в data-файле, где это value лежит.
А `val_len` нужен, чтобы при чтении знать, сколько байт считать по этому смещению.

Насколько это экономит? Допустим, data-файл на 10 ГБ, средний размер value — 1 КБ, средний ключ — 20 байт.
В data-файле ~10 миллионов записей. Без hint-файла при восстановлении нужно прочитать все 10 ГБ.
Hint-файл для тех же записей содержит только метаданные (~40 байт на запись): примерно 400 МБ — в 25 раз меньше.
Чем крупнее значения — тем больше выигрыш.

При старте логика восстановления выглядит так:

```
  Для каждого data-файла:
    1. Есть hint-файл?
       ├─ Да  → читаем hint-файл (маленький, только ключи + offsets)
       │        для каждой записи: index[key] = {file_id, val_pos, val_len}
       │
       └─ Нет → читаем data-файл целиком (большой, ключи + значения)
                для каждой записи: index[key] = {file_id, offset, val_len}
```

Hint-файлы создаются только при compaction, а не при обычной записи — потому что активный файл всё равно меняется
и его hint устареет. После merge старые data-файлы и их hint-файлы удаляются, остаётся один
компактный data-файл + один hint-файл.

В нашей реализации мы hint-файлы не делаем — у нас один файл и нет compaction. Но сама 
идея «хранить метаданные отдельно от данных» появится снова, когда мы дойдём до SSTable.

## Сколько памяти занимает индекс

Это ключевой вопрос, потому что **все ключи должны поместиться в RAM**. Давайте посчитаем.

В Go `map[string]int64` для каждой записи хранит:

- **Ключ (string):** 16 байт на string header (pointer + length) + длина самой строки
- **Значение (int64):** 8 байт
- **Overhead hash table:** ~30-50 байт на запись (bucket metadata, hash values, указатели, padding)

Для ключа средней длины 20 байт получается примерно **70-90 байт на запись**.

```
  Оценка потребления памяти для разного количества ключей:

  Ключей          Память (при ~80 байт/ключ)
  ─────────────── ─────────────────────────────
  100 000              ~8 MB
  1 000 000           ~80 MB
  10 000 000         ~800 MB
  100 000 000          ~8 GB
  1 000 000 000       ~80 GB
```

Для 100 миллионов ключей индекс занимает порядка 8 ГБ оперативной памяти. Это серьёзное ограничение.
При этом сами значения могут лежать на диске терабайтами — проблема не в объёме данных, а именно в количестве уникальных ключей.

## Сложность операций

| Операция | Было (append-only) | Стало (с hash index) | Комментарий |
|----------|:-------------------:|:--------------------:|-------------|
| Set      | O(1)                | O(1)                 | Append в файл + обновление hash table |
| Get      | O(n)                | O(1)                 | Один lookup + один disk read |
| Delete   | O(1)                | O(1)                 | Append tombstone + delete из hash table |

## Плюсы

- **Быстрое чтение и запись.** Обе операции — O(1). Запись — sequential append. Чтение — один random read с диска (один disk seek).
- **Простота.** Реализация добавляет буквально одно поле (`map[string]int64`) и несколько строк кода к тому, что уже есть.
- **Данные больше RAM.** Значения хранятся на диске — суммарный объём данных может многократно превышать объём оперативной памяти. В RAM хранятся только ключи.

## Минусы

- **Все ключи должны поместиться в RAM.** Это фундаментальное ограничение. Если у вас миллиард ключей — нужны десятки гигабайт только под индекс. Для многих сценариев это приемлемо, для некоторых — нет.
- **Нет range scan.** Hash table не умеет отвечать на запрос «дай мне все ключи от A до Z» — нет упорядоченности. Это не артефакт реализации, а фундаментальное свойство hash table. Для этого нужна другая структура данных.
- **Файл растёт бесконечно.** Мы по-прежнему не удаляем устаревшие записи. Файл только растёт. Нужен compaction, но в этой главе мы его не рассматриваем.
- **Восстановление при старте O(n).** Каждый перезапуск — полный скан файла для восстановления индекса.
- **Один random read на каждый Get.** Для SSD это микросекунды, для HDD — миллисекунды. При высокой нагрузке на чтение это может стать узким местом.

## Почему range scan — это важно

Может показаться, что отсутствие range scan — мелочь. Но на практике огромное количество запросов в базах данных — это именно range query:

```sql
  -- все заказы пользователя за последний месяц
SELECT * FROM orders
WHERE user_id = 42
  AND created_at > '2026-04-01';

-- все ключи с определённым префиксом
SCAN "user:1000:*"

  -- top-N по какому-то критерию
SELECT * FROM products
ORDER BY price ASC
    LIMIT 10;
```

Для hash table каждый такой запрос — это полный перебор всех ключей.

```go
	// Портим данные после построения индекса (индекс уже указывает на запись).
	// Перезаписываем value-часть записи, не трогая заголовок.
	f, _ := os.OpenFile(path, os.O_RDWR, 0644)
	// recordHeaderSize(12) + len("key")(3) = 15 — начало value
	f.WriteAt([]byte("XXXXX"), 15)
	f.Close()

	// Get должен обнаружить CRC mismatch.
	_, err := db.Get([]byte("key"))
	if err != ErrCorrupted {
		t.Fatalf("expected ErrCorrupted, got %v", err)
	}
	db.Close()
}

func BenchmarkHashDBGet(b *testing.B) {
	db, _ := OpenHashDB(filepath.Join(b.TempDir(), "bench.log"))
	// Наполняем 100k ключей:
	for i := 0; i < 100_000; i++ {
		key := fmt.Appendf(nil, "key-%d", i)
		db.Set(key, []byte("value"))
	}

	b.ResetTimer()
	b.RunParallel(func(pb *testing.PB) {
		i := 0
		for pb.Next() {
			key := fmt.Appendf(nil, "key-%d", i%100_000)
			db.Get(key)
			i++
		}
	})
}

```


Обратите внимание на несколько моментов:

- **`valueLocation` хранит `offset` и `size`** — это позволяет читать ровно столько байт, сколько нужно, через `ReadAt`, без предварительного парсинга заголовка.
- **`buildIndex` требует доработки `Log.scan`** — текущая версия `scan` не передаёт offset записи в callback. Нужно либо расширить callback, либо добавить отдельный метод.
- **`Stats`** — полезная функция для понимания, сколько мусора накопилось в файле. `SpaceAmplification = TotalFileBytes / LiveDataBytes` показывает, во сколько раз файл больше, чем живые данные. Когда этот коэффициент становится слишком большим — пора делать compaction.
