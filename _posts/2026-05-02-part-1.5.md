В предыдущей главе мы разобрали архитектуру LSM-tree и увидели, что все записи сначала попадают в memtable — отсортированную структуру данных в памяти. Memtable должна уметь три вещи: быстро вставлять, быстро искать и отдавать данные в отсортированном порядке (для flush в SSTable и для range scan).

Hash table, которую мы использовали раньше, не подходит — она не поддерживает упорядоченный обход. Нужна другая структура.

## Почему не hash table

В главе 1.2 мы использовали `map[string]valueLocation` — hash table, которая давала O(1) на чтение и запись. Отличная производительность, но для memtable в LSM-tree hash table не годится по двум причинам:

**Нет упорядоченного обхода.** Когда memtable заполняется, её содержимое нужно записать на диск в отсортированном порядке — это формат SSTable. Hash table не хранит порядок ключей. Чтобы получить отсортированные данные, пришлось бы собирать все ключи, сортировать их (O(n log n)) и только потом записывать. Skip list отдаёт данные в отсортированном порядке за O(n) — просто проход по нижнему уровню.

**Нет range scan.** Запрос «дай все ключи от `user:100` до `user:200`» требует полного перебора hash table. В skip list — это поиск начальной позиции за O(log n) и последовательный проход.

| | Hash Table | Skip List |
|---|---|---|
| **Поиск по ключу** | O(1) | O(log n) |
| **Вставка** | O(1) | O(log n) |
| **Упорядоченный обход** | O(n log n) — собрать и отсортировать | O(n) — пройти по уровню 0 |
| **Range scan** | O(n) — полный перебор | O(log n + k) — поиск + проход |
| **Flush в SSTable** | Требует сортировки | Данные уже отсортированы |

Skip list проигрывает hash table на точечных операциях (O(log n) vs O(1)), но для memtable это приемлемо: записи накапливаются в памяти и периодически сбрасываются на диск, а не живут в памяти вечно. Зато упорядоченность данных даёт нам range scan и бесплатный flush.

## Почему именно skip list

Skip list — вероятностная структура данных на основе связных списков. O(log n) в среднем на вставку и поиск, поддерживает упорядоченный обход. Реализуется за ~100 строк Go.

Для memtable в LSM-tree у skip list есть три ключевых преимущества:

**Простота.** Skip list реализуется за сотню строк, и каждая из них понятна. Для storage engine, где надёжность критична, простота кода — серьёзное преимущество. Меньше кода — меньше багов — проще поддерживать.

**Параллельный доступ.** В memtable одновременно пишут и читают. В skip list вставка нового узла — локальная операция: нужно обновить несколько указателей, не затрагивая остальную структуру. Это делает конкурентные реализации значительно проще.

**Эффективный flush.** Когда memtable заполняется, нужно записать все данные на диск в отсортированном порядке. В skip list — это проход по нижнему уровню, O(n). Просто и предсказуемо.

Именно поэтому LevelDB выбрал skip list для своей memtable. В оригинальной статье Уильяма Пью 1990 года skip list позиционируется как «вероятностная альтернатива сбалансированным деревьям» — и для memtable эта альтернатива оказалась идеальной. RocksDB, Pebble, HBase, Redis — все пошли тем же путём.

## Как устроен skip list

Чтобы понять skip list, начнём с самого простого — со связного списка — и будем постепенно улучшать его, пока не получим skip list.

### Шаг 0: что такое связный список

Связный список — это цепочка узлов, где каждый узел хранит значение и ссылку на следующий узел. В отличие от массива, узлы не лежат в памяти подряд — каждый может быть где угодно, а связь между ними обеспечивается указателями.

```
  ┌───────────┐    ┌───────────┐    ┌───────────┐    ┌───────────┐
  │ key: 3    │    │ key: 7    │    │ key: 12   │    │ key: 19   │
  │ value: a  │───►│ value: b  │───►│ value: c  │───►│ value: d  │───► NIL
  └───────────┘    └───────────┘    └───────────┘    └───────────┘
```

Если список отсортирован по ключу, мы можем искать элементы последовательно: начинаем с головы и идём вправо, пока не найдём нужный ключ или не дойдём до конца.

### Шаг 1: один уровень — отсортированный связный список

Возьмём отсортированный связный список из 8 элементов:

```
  Level 0:  HEAD → 3 → 7 → 12 → 19 → 26 → 35 → 42 → 50 → NIL
```

Хотим найти ключ 42. Начинаем с HEAD и идём вправо: 3, 7, 12, 19, 26, 35, 42 — нашли! Потребовалось 7 шагов. Для ключа 50 — 8 шагов. Это O(n), ровно та же проблема, что была у нашего append-only log.

Плюс: вставка в нужную позицию — O(1) после того, как мы нашли место (просто переставляем указатели). Данные всегда отсортированы, обход — тривиальный.

Минус: поиск — O(n). Для memtable с миллионами записей это слишком медленно.

### Шаг 2: добавляем второй уровень

Поиск O(n) — это медленно. Как ускорить? Идея: а что если завести **второй, более короткий список**, в который попадут не все узлы, а только некоторые — например, каждый четвёртый? И каждый узел в этом коротком списке будет хранить указатель вниз на тот же узел в основном списке.

Получается два списка: нижний (level 0) содержит все элементы, верхний (level 1) — только избранные:

```
  Level 1:  HEAD ──────────── 12 ──────────────── 35 ────────── NIL
                               │                   │
                          указатель вниз       указатель вниз
                               │                   │
  Level 0:  HEAD → 3 → 7 → 12 → 19 → 26 → 35 → 42 → 50 → NIL
```

Теперь поиск выглядит иначе. Сначала идём по верхнему (короткому) списку — это быстро, потому что в нём мало узлов. Как только увидели, что следующий узел на верхнем уровне больше искомого — спускаемся вниз и доходим по нижнему списку.

Ищем ключ 42:
1. Level 1: HEAD → 12 (12 < 42, идём дальше) → 35 (35 < 42, идём дальше) → NIL. Дошли до конца верхнего списка, значит 42 где-то после 35. Спускаемся вниз с узла 35.
2. Level 0: 35 → 42 — нашли!

Всего 4 шага вместо 7. Верхний уровень позволил перепрыгнуть через 3, 7 и через 19, 26 — мы их даже не смотрели.

Ищем ключ 19:
1. Level 1: HEAD → 12 (12 < 19, идём дальше) → 35 (35 > 19, стоп!). Спускаемся вниз с узла 12.
2. Level 0: 12 → 19 — нашли!

Всего 3 шага. Верхний список сработал как оглавление: мы быстро определили, что 19 находится между 12 и 35, и сразу перешли в нужный участок нижнего списка.

Важно: `12` на level 1 и `12` на level 0 — это **один и тот же узел** в памяти. Просто у него два указателя `next`: один ведёт к следующему узлу на level 0, другой — на level 1. «Спуститься вниз» означает переключиться с `next[1]` на `next[0]` того же узла.

### Шаг 3: добавляем третий уровень

Добавим ещё один «экспресс-уровень» поверх второго:

```
  Level 2:  HEAD ──────────────────────────────── 35 ────────── NIL
              │                                    │
  Level 1:  HEAD ──────────── 12 ──────────────── 35 ────────── NIL
              │                │                   │
  Level 0:  HEAD → 3 → 7 → 12 → 19 → 26 → 35 → 42 → 50 → NIL
```

Ищем ключ 19:
1. Level 2: HEAD → 35. 35 > 19, спускаемся с HEAD.
2. Level 1: HEAD → 12. 12 < 19, идём дальше. 12 → 35. 35 > 19, спускаемся с 12.
3. Level 0: 12 → 19 — нашли!

3 шага. Каждый новый уровень сокращает поиск примерно вдвое.

### Шаг 4: четыре уровня — полноценный skip list

```
  Level 3:  HEAD ─────────────────────────────── 35 ────────── NIL
              │                                   │
  Level 2:  HEAD ──────────── 12 ──────────────── 35 ────────── NIL
              │                │                   │
  Level 1:  HEAD ───── 7 ──── 12 ──────── 26 ──── 35 ────────── NIL
              │        │       │           │       │
  Level 0:  HEAD → 3 → 7 → 12 → 19 → 26 → 35 → 42 → 50 → NIL
```

Теперь ищем ключ 26:
1. Level 3: HEAD → 35. 35 > 26, спускаемся.
2. Level 2: HEAD → 12. 12 < 26, идём. 12 → 35. 35 > 26, спускаемся с 12.
3. Level 1: 12 → 26 — нашли!

3 шага для 8 элементов. Это O(log n) — на каждом уровне мы отсекаем примерно половину оставшихся элементов.

### Главное: узлы — не копии

На диаграмме `12` появляется на уровнях 0, 1, 2, а `35` — на уровнях 0, 1, 2, 3. Но это **не копии данных**. Это **один и тот же узел** в памяти. Каждый узел хранит массив указателей `next[]` — по одному на каждый уровень, на котором он присутствует:

```
  Узел с ключом 12 (высота 3):

  ┌──────────────────────┐
  │  key:   []byte("12") │  ← ключ, по которому ищем
  │  value: []byte("v1") │  ← значение, видимое со всех уровней
  │                      │
  │  next[2]: ───────────────► узел 35   (уровень 2)
  │  next[1]: ───────────────► узел 26   (уровень 1)
  │  next[0]: ───────────────► узел 19   (уровень 0)
  └──────────────────────┘

  Уровни — это не отдельные списки.
  Уровни — это разные «дорожки» через одни и те же узлы,
  реализованные через массив указателей next[].

  Если вызвать Set("12", "v2"), обновится поле value
  в этом единственном узле — и новое значение сразу
  видно при поиске с любого уровня.
```

### Как определяется высота узла

При вставке нового узла его высота выбирается случайно. С вероятностью 1/4 (наш `branchFactor = 4`) узел поднимается на каждый следующий уровень. Это похоже на подбрасывание нечестной монеты: «орёл» (вероятность 1/4) — добавляем ещё один уровень, «решка» (вероятность 3/4) — останавливаемся.

В итоге большинство узлов остаются на уровне 0, и лишь немногие поднимаются высоко. Это и создаёт структуру «экспресс-дорожек» естественным образом, без какой-либо ручной балансировки.

## Устройство нашего skip list

Каждый узел хранит ключ, значение и массив указателей `next` — по одному на каждый уровень, на котором узел присутствует:

```
  type skipListNode struct {
      key   []byte
      value []byte
      next  [maxHeight]*skipListNode  // указатели на следующий узел на каждом уровне
  }
```

Конкретные числа по распределению высот при `branchFactor = 4`:

```
  Распределение высот (branchFactor = 4):

  Высота 1:  75.0%   ████████████████████████████████████
  Высота 2:  18.75%  █████████
  Высота 3:   4.69%  ██
  Высота 4:   1.17%  █
  Высота 5:   0.29%  ▏
  ...
  Высота 20:  ≈0%    (maxHeight, практически недостижимо)
```

Структура skip list в целом:

```
  type SkipList struct {
      head     skipListNode  // фиктивный головной узел (без ключа)
      height   int           // текущая максимальная высота
      length   int           // количество элементов
      byteSize int64         // суммарный размер ключей + значений
  }
```

`head` — фиктивный узел, от которого начинаются все уровни. `byteSize` нужен, чтобы определить момент, когда memtable заполнилась и пора сбрасывать на диск.

## Операция Set: вставка и обновление

Вставка — самая сложная операция. Она состоит из четырёх шагов.

**Шаг 1: найти позицию вставки на каждом уровне.** Спускаемся от верхнего уровня к нижнему, на каждом уровне двигаемся вправо, пока следующий ключ меньше вставляемого. Запоминаем последний узел на каждом уровне в массив `update[]` — это те узлы, чьи указатели нужно будет обновить.

```
  Set(key=20) в skip list:

  Level 2:  HEAD ──────────── 12 ──────────── 35 ── NIL
                              ▲ update[2]
  Level 1:  HEAD ───── 7 ──── 12 ──────────── 35 ── NIL
                              ▲ update[1]
  Level 0:  HEAD → 3 → 7 → 12 → 19 → 26 → 35 → NIL
                              ▲ update[0] = 19
                                   ^
                                   Вставляем 20 сюда
```

Подождите — `update[0]` должен быть узлом 19, потому что 19 < 20 < 26. На уровнях 1 и 2 последний узел с ключом < 20 — это 12.

**Шаг 2: проверить, не существует ли ключ.** Если `update[0].next[0].key == key`, то ключ уже есть — обновляем значение и выходим.

**Шаг 3: выбрать случайную высоту** для нового узла с помощью `randomHeight()`.

**Шаг 4: вставить узел,** обновив указатели на каждом уровне:

```
  randomHeight() вернул 2. Вставляем узел 20 на уровнях 0 и 1:

  До:
  Level 1:  ... → 12 ──────────────── 35 → ...
  Level 0:  ... → 19 ──────── 26 → ...

  После:
  Level 1:  ... → 12 ────── 20 ────── 35 → ...
  Level 0:  ... → 19 → 20 → 26 → ...

  На каждом уровне i:
    newNode.next[i] = update[i].next[i]   // новый узел указывает на следующий
    update[i].next[i] = newNode            // предыдущий узел указывает на новый
```

## Операция Get: поиск по ключу

Поиск проще: спускаемся от верхнего уровня к нижнему, на каждом уровне двигаемся вправо, пока можем. На уровне 0 проверяем, совпадает ли ключ.

```
  Get(key=26):

  Level 2:  HEAD ──────────── 12 ──────────── 35     35 > 26, спускаемся
                              ▲ здесь
  Level 1:  ................. 12 ──── 20 ──── 35     35 > 26, спускаемся
                                     ▲ здесь
  Level 0:  ................. 19 → 20 → 26           26 == 26, нашли!
                                        ▲
```

Если на уровне 0 следующий ключ != искомому — ключа нет в skip list.

## Операция Delete: tombstone

В контексте LSM-tree удаление — это вставка со значением `nil`. Когда memtable сбрасывается в SSTable, записи с `nil` value становятся tombstone-маркерами. При compaction tombstone удаляет соответствующий ключ из старых SSTable.

```go
func (sl *SkipList) Delete(key []byte) {
    sl.Set(key, nil) // nil value = tombstone
}
```

Просто и элегантно — не нужна отдельная логика удаления из структуры.

## Итератор: последовательный обход

Для flush memtable в SSTable нужно пройти по всем ключам в отсортированном порядке. На уровне 0 все элементы связаны в отсортированный список — просто идём по нему:

```
  Level 0:  HEAD → 3 → 7 → 12 → 19 → 20 → 26 → 35 → 42 → NIL
                   ▲
                   итератор начинает здесь (head.next[0])
                   и идёт по next[0] до NIL
```

Итератор также поддерживает `SeekGE(target)` — перемещение к первому ключу >= target. Это нужно для range scan (`SELECT * WHERE key >= 'user:100'`) и для merge при compaction. `SeekGE` использует уровни skip list для быстрого позиционирования — O(log n), а не O(n).

## Когда skip list проигрывает

Skip list — не идеальная структура. Стоит знать её слабые места:

**Локальность кэша.** Каждый узел — отдельная аллокация в куче. Указатели прыгают по памяти хаотично, что плохо для кэша процессора. В production-реализациях это решают через arena-аллокаторы: узлы выделяются в одном непрерывном блоке памяти (так сделано в LevelDB).

**Вероятностные гарантии.** O(log n) — это среднее, а не гарантированное. Теоретически возможен worst case O(n), если генератор случайных чисел выдаст всем узлам высоту 1. На практике это исчезающе маловероятно, но формально гарантии нет.

**Расход памяти.** Каждый узел хранит массив указателей (в нашем случае — до 20 штук, хотя большинство использует 1-2). При `branchFactor = 4` средний расход — 1.33 указателя на узел, что приемлемо.

## Выводы

Skip list — это компромисс, который оказался идеальным именно для memtable в LSM-tree:

- O(log n) на вставку и поиск — достаточно быстро для буфера в памяти
- O(n) последовательный обход — именно то, что нужно для flush в SSTable
- Простая реализация — меньше багов, проще поддерживать
- Хорошая совместимость с параллельным доступом — вставка не затрагивает глобальную структуру
- Tombstone через `nil` value — удаление без модификации структуры

LevelDB, RocksDB, Pebble, HBase, Redis — все выбрали skip list для in-memory отсортированных структур. Это не случайность, а результат того, что простота и практичность перевесили теоретическое превосходство сбалансированных деревьев.

На первый взгляд skip list может показаться сложной структурой — лично я потратил пару дней, чтобы разобраться в деталях: как работают уровни, как обновляются указатели при вставке, как правильно реализовать поиск с «проваливанием» между уровнями. Если вы никогда не работали со связными списками, рекомендую сначала порешать задачи на [LeetCode](https://leetcode.com/tag/linked-list/) по теме linked list и почитать про паттерн **фиктивный узел** (dummy head / sentinel node) — это именно то, что мы используем в нашем skip list как `head`. Фиктивный узел упрощает граничные случаи: не нужно отдельно обрабатывать вставку в начало списка, потому что `head` всегда существует и никогда не удаляется.

## Задание

Реализуйте skip list на Go.

### Шаблон

```go
package storage

import (
    "bytes"
    "math/rand"
    "sync"
)

const (
    maxHeight    = 20
    branchFactor = 4 // вероятность подняться на уровень выше = 1/branchFactor
)

type skipListNode struct {
    key   []byte
    value []byte
    next  [maxHeight]*skipListNode
}

type SkipList struct {
    mu       sync.RWMutex
    head     skipListNode
    height   int     // текущая максимальная высота
    length   int
    byteSize int64   // суммарный размер ключей + значений (для flush threshold)
}

func NewSkipList() *SkipList {
    return &SkipList{height: 1}
}

// Set вставляет или обновляет ключ.
func (sl *SkipList) Set(key, value []byte) {
    sl.mu.Lock()
    defer sl.mu.Unlock()

    // TODO:
    //   1. Найти позицию вставки на каждом уровне (массив update[]).
    //      Для каждого уровня от высокого к низкому:
    //        двигайся вправо пока next[level].key < key
    //        запоминай последний узел на этом уровне в update[level]
    //   2. Проверить: если update[0].next[0].key == key → обновляем value, выходим
    //      Не забудь скопировать value (append([]byte{}, value...)),
    //      иначе caller сможет мутировать данные внутри skip list.
    //      Обнови byteSize при перезаписи.

func TestSkipListPropertyBased(t *testing.T) {
	// Property-based тест без внешних зависимостей: генерируем случайные операции
	// и сверяем SkipList с эталонной map на каждом шаге.
	rng := rand.New(rand.NewSource(42))

	const (
		iterations = 5000
		keySpace   = 200
	)

	sl := NewSkipList()
	ref := make(map[string]string)

	for i := 0; i < iterations; i++ {
		key := fmt.Sprintf("k%03d", rng.Intn(keySpace))
		val := fmt.Sprintf("v%d", rng.Intn(1000))

		op := rng.Intn(3)
		switch op {
		case 0:
			sl.Set([]byte(key), []byte(val))
			ref[key] = val

		case 1:
			sl.Delete([]byte(key))
			delete(ref, key)

		case 2:
			slVal, slOK := sl.Get([]byte(key))
			refVal, refOK := ref[key]

			slActualOK := slOK && slVal != nil

			if slActualOK != refOK {

					i, key, slActualOK, refOK)

				t.Fatalf("op %d: Get(%q): skiplist=%q, map=%q",
					i, key, slVal, refVal)
			}
		}
	}

	it := sl.NewIterator()
	var slKeys []string
	for ; it.Valid(); it.Next() {
		if it.Value() != nil {
			k := string(it.Key())
			slVal := string(it.Value())
			refVal, ok := ref[k]
			if !ok {
				t.Fatalf("skiplist has key %q not in reference map", k)
			}
			if slVal != refVal {
				t.Fatalf("key %q: skiplist=%q, map=%q", k, slVal, refVal)
			}
			slKeys = append(slKeys, k)
		}
	}

	if len(slKeys) != len(ref) {
		t.Fatalf("skiplist has %d keys, map has %d keys", len(slKeys), len(ref))
	}

	for i := 1; i < len(slKeys); i++ {
		if slKeys[i] <= slKeys[i-1] {
			t.Fatalf("keys not sorted at position %d: %q <= %q",
				i, slKeys[i], slKeys[i-1])
		}
	}
}

func TestSkipListLevelDistribution(t *testing.T) {
	// В правильном skiplist на верхних уровнях должно быть существенно
	// меньше узлов чем на нижних (1/branchFactor на каждый уровень вверх).
	sl := NewSkipList()
	const n = 10_000
	for i := 0; i < n; i++ {
		sl.Set(fmt.Appendf(nil, "k%06d", i), []byte("v"))
	}

	// Считаем узлы на каждом уровне:
	counts := make([]int, sl.height)
	for i := 0; i < sl.height; i++ {
		curr := sl.head.next[i]
		for curr != nil {
			counts[i]++
			curr = curr.next[i]
		}
	}

	// Уровень 0 содержит все узлы:
	if counts[0] != n {
		t.Fatalf("level 0: got %d nodes, want %d", counts[0], n)
	}

	// Каждый следующий уровень должен быть существенно меньше.
	// При branchFactor=4 ожидаем ratio ~0.25. Допускаем большой разброс,
	// но каждый верхний уровень должен быть <= 60% от нижнего.
	for i := 1; i < sl.height; i++ {
		if counts[i] == 0 {
			break // достигли пустых верхних уровней — норм
		}
		ratio := float64(counts[i]) / float64(counts[i-1])
		if ratio > 0.6 {
			t.Fatalf("level %d has %d nodes (%.1f%% of level %d=%d); "+
				"expected ~25%% — skiplist degenerated to linked list?",
				i, counts[i], ratio*100, i-1, counts[i-1])
		}
	}
	t.Logf("level distribution: %v", counts)
}

				for j := 0; j < n; j++ {
					m[string(keys[j])] = []byte("v")
				}
			}
		})

		sl := NewSkipList()
		m := make(map[string][]byte, n)
		for j := 0; j < n; j++ {
			sl.Set(keys[j], []byte("v"))
			m[string(keys[j])] = []byte("v")
		}

		b.Run(fmt.Sprintf("SkipList/Get/n=%d", n), func(b *testing.B) {
			b.ResetTimer()
			for i := 0; i < b.N; i++ {
				sl.Get(keys[i%n])
			}
		})

		b.Run(fmt.Sprintf("Map/Get/n=%d", n), func(b *testing.B) {
			b.ResetTimer()
			for i := 0; i < b.N; i++ {
				_ = m[string(keys[i%n])]
			}
		})

		scanStart := keys[n/4]
		scanEnd := keys[n/2]

		b.Run(fmt.Sprintf("SkipList/RangeScan/n=%d", n), func(b *testing.B) {
			b.ResetTimer()
			for i := 0; i < b.N; i++ {
				sl.RangeScan(scanStart, scanEnd)
			}
		})

		b.Run(fmt.Sprintf("Map/RangeScan/n=%d", n), func(b *testing.B) {
			b.ResetTimer()
			for i := 0; i < b.N; i++ {
				var result [][]byte
				for k := range m {
					kb := []byte(k)
					if bytes.Compare(kb, scanStart) >= 0 && bytes.Compare(kb, scanEnd) < 0 {
						result = append(result, kb)
					}
				}
				sort.Slice(result, func(i, j int) bool {
					return bytes.Compare(result[i], result[j]) < 0
				})
			}
		})
	}
}
```

Обратите внимание на несколько моментов:

- **`maxHeight = 20`** — достаточно для ~4^20 ≈ 10^12 элементов. На практике memtable содержит миллионы записей — высоты 20 более чем достаточно.
- **`branchFactor = 4`** — вероятность подъёма 1/4. Это означает меньше уровней, чем при классическом 1/2 (как в оригинальной статье Пью), но лучшее соотношение памяти к скорости. LevelDB использует тот же множитель.
- **`byteSize`** — суммарный размер ключей и значений. Используется для определения момента flush: когда `byteSize` превышает порог (обычно 4-64 МБ), memtable сбрасывается на диск.
- **`Delete` = `Set(key, nil)`** — tombstone реализован через nil value, без модификации структуры skip list.
- **`SeekGE`** — важная операция для range scan. Использует уровни skip list для быстрого позиционирования, а не линейный проход по уровню 0.
